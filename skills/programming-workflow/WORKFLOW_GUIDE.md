### 1.2 核心原则

| 原则 | 说明 |
|------|------|
| **智能灵活** | 阶段可并行、可回溯、可跳过，根据项目实际情况调整 |
| **技能驱动** | 每个阶段有对应的 Skills 支持，提高效率 |
| **协作优先** | 多 Agent 协同，明确分工，高效配合 |
| **进度透明** | 任务看板 + 每日日志 + Git 提交记录 |
| **质量保障** | 测试门禁 + 代码评审 + 产品验收 |

## 三、进度同步机制

### 3.1 任务看板

**位置**: `~/.openclaw/ai-chat-room/tasks/<项目名>.md`

**状态标识**:
- 📋 待办 (Todo)
- 🔄 进行中 (In Progress)
- ✅ 完成 (Done)
- ❌ 阻塞 (Blocked)
- ⏸️ 暂停 (Paused)

**看板格式**:
```markdown
# 项目任务看板

## 📋 待办
- [ ] 任务1 (P0) - 负责人
- [ ] 任务2 (P1) - 负责人

## 🔄 进行中
- [ ] 任务3 (P0) - 小琳 - 预计完成: 2026-03-15

## ✅ 完成
- [x] 任务4 (P0) - 小猪 - 完成于: 2026-03-12

## ❌ 阻塞
- [ ] 任务5 (P1) - 小琳 - 阻塞原因: 等待设计确认
```

### 3.2 每日日志

**位置**: `memory/YYYY-MM-DD.md`

**格式**:
```markdown
# 2026-03-12 今日记录

## 完成任务
- [x] TypeScript 迁移完成
- [x] API 接口开发

## 进行中
- [ ] 前端页面开发 (进度 60%)

## 问题与阻塞
- Redis 连接不稳定，需要排查

## 明日计划
```

### 3.3 进度同步频率

| 场景 | 频率 | 方式 |
|------|------|------|
| **日常开发** | 每日更新 | 每日日志 + 任务看板 |

## 四、版本迭代管理

### 4.1 版本号规范

采用语义化版本: `MAJOR.MINOR.PATCH`

| 版本类型 | 说明 | 示例 |
|----------|------|------|
| **MAJOR** | 重大更新，不兼容变更 | 1.0.0 → 2.0.0 |
| **MINOR** | 功能新增，向后兼容 | 1.0.0 → 1.1.0 |
| **PATCH** | Bug 修复，向后兼容 | 1.0.0 → 1.0.1 |

### 4.2 分支策略

```
master (生产环境)
  ↑
dev (开发环境)
  ↑
feat-<username>-<feature>-<version> (功能分支)
```

**分支命名示例**:
- `feat-xiaolin-chat-ui-1.0.0`
- `feat-xiaozhu-agent-api-1.0.0`
- `fix-xiaolin-login-bug-1.0.1`

### 4.3 发布流程

```
开发完成 → 合并 dev → 测试 → 合并 master → 打 Tag → 发布
    │           │        │           │           │        │
    ▼           ▼        ▼           ▼           ▼        ▼
 功能分支    PR Review  QA验证    Code Review  Git Tag  部署
```

### 4.4 CHANGELOG 维护

**位置**: `CHANGELOG.md`

**格式**:
```markdown
# Changelog

## [1.1.0] - 2026-03-15

### Added
- 新增 Agent 管理功能
- 新增消息搜索功能

### Changed
- 优化登录页面 UI

### Fixed
- 修复 WebSocket 重连问题

### Security
- 修复 XSS 漏洞
```

## 五、多 Agent 协同机制

### 5.1 角色分工

| Agent | 主要职责 | 技能特长 |
|-------|----------|----------|
| **小琳** (lin-bot) | 前端开发、UI 设计、用户体验 | Vue、React、UI/UX |
| **小猪** (maple-bot) | 后端开发、API 设计、性能优化 | Node.js、数据库、运维 |
| **小熊** (bear-bot) | 全栈开发、测试、文档 | 全栈技术、测试框架 |

### 5.2 协作通信

**通信方式**: Git 仓库共享 + 钉钉群消息

**消息格式**:
```
@<Agent名> <任务描述>

示例:
@小琳 请完成登录页面的 UI 开发
@小猪 需要新增一个 API 接口: /api/users/profile
```

### 5.3 任务分配原则

| 原则 | 说明 |
|------|------|
| **专长优先** | 根据技能特长分配任务 |
| **负载均衡** | 避免某个 Agent 过载 |
| **并行开发** | 独立任务可并行进行 |
| **及时同步** | 有依赖的任务需提前沟通 |

### 5.4 冲突解决

**代码冲突**:
```
git pull --rebase → 解决冲突 → git push
```

**任务冲突**:
- 通过任务看板明确任务归属
- 使用文件名后缀区分（如 `xiaolin-xxx.ts`、`xiaozhu-xxx.ts`）

**决策冲突**:
- 技术决策: 小猪主导
- 产品决策: maple 确认
- 设计决策: 小琳主导

## 六、智能灵活协调

### 6.1 阶段并行

**可并行阶段**:
- 需求整理 + 技术调研
- UI 设计 + API 设计
- 前端开发 + 后端开发
- 测试 + 文档编写

**示例**:
```
Day 1: 需求整理 (小琳) + 技术调研 (小猪) - 并行
Day 2: UI 设计 (小琳) + API 设计 (小猪) - 并行
Day 3-5: 前端开发 (小琳) + 后端开发 (小猪) - 并行
```

### 6.2 阶段跳过

**可跳过场景**:
- 小改动：需求 → 开发 → 测试 → 上线
- Bug 修复：开发 → 测试 → 上线
- 文档更新：直接修改 → 发布

### 6.3 阶段回溯

**回溯触发条件**:
- 需求变更 → 回到需求整理
- 设计问题 → 回到 UI 设计
- 技术方案调整 → 回到架构规划

### 6.4 灵活组合示例

**场景 1: 新功能开发**
```
需求整理 → 产品设计 → UI设计 → 并行开发 → 测试 → 验收 → 上线
```

**场景 2: Bug 修复**
```
问题确认 → 开发修复 → 测试 → 验证 → 发布
```

**场景 3: 快速原型**
```
需求 → 快速开发 → 内部测试 → 迭代优化
```

**场景 4: 技术债务清理**
```
问题分析 → 方案设计 → 重构 → 测试 → 发布
```

## 七、质量门禁

### 7.1 代码提交门禁

| 检查项 | 要求 | 工具 |
|--------|------|------|
| 代码风格 | ESLint 通过 | ESLint |
| 类型检查 | TypeScript 无错误 | tsc |
| 提交信息 | Conventional Commits | commitlint |

### 7.2 合并门禁

| 检查项 | 要求 |
|--------|------|
| 代码评审 | 至少 1 人 Approve |
| 测试通过 | 所有测试用例通过 |
