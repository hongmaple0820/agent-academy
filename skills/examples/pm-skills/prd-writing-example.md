# PM Skills 使用示例

> 使用 PM 技能包完成产品需求文档 (PRD) 写作

## 📋 场景说明

**场景**: 有一个新产品想法，需要写成正式的 PRD
**技能**: pm-skills 的 `/write-prd` 命令
**前置**: 已完成 brainstorming 需求探索

---

## 💬 完整对话示例

### 第一步：启动 PRD 写作

```
用户：需求已经明确了，帮我写 PRD。

小熊：好的！使用 pm-skills 的 `/write-prd` 技能。🐻

📝 **PRD 结构**

根据 PM 最佳实践，PRD 包含：
1. 产品概述
2. 目标用户
3. 功能列表
4. 用户故事
5. 验收标准
6. 非功能需求
7. 发布标准

请告诉我产品的核心信息：
- 产品名称是什么？
- 解决什么问题？
- 目标用户是谁？
```

### 第二步：填写产品概述

```
用户：产品叫"知识库 API"，用于 AI 助手共享和检索知识文档。

小熊：✅ **1. 产品概述**

**产品名称**: 知识库 API (Knowledge Base API)

**产品愿景**: 
让 AI 助手能够高效地共享、检索和更新知识文档，
建立统一的协作知识平台。

**解决的问题**:
- 知识分散，难以共享
- 检索效率低
- 版本管理混乱

**核心价值**:
- 统一的知识存储
- 高效的全文搜索
- 清晰的版本管理

继续下一步，目标用户是谁？
```

### 第三步：定义目标用户

```
用户：主要用户是小琳、小猪、小熊三个 AI 助手，还有人类管理员 maple。

小熊：✅ **2. 目标用户**

**主要用户**:

|------|------|---------|
| 小琳 | AI 助手 | 查询前端相关知识 |
| 小猪 | AI 助手 | 查询后端相关知识 |
| 小熊 | AI 助手 | 查询运维相关知识 |
| maple | 管理员 | 审核和管理文档 |

**用户特征**:
- 技术背景强
- 需要快速检索
- 高并发访问

继续，核心功能有哪些？
```

### 第四步：功能列表

```

**P0 - 核心功能** (MVP 必须)
- [ ] 文档创建 (Create)

- [ ] 文档更新 (Update)
- [ ] 文档删除 (Delete)
- [ ] 全文搜索 (FTS5)

**P1 - 重要功能** (v1.1)
- [ ] 版本管理
- [ ] 权限控制
- [ ] 变更日志

**P2 - 增强功能** (v1.2+)
- [ ] 文档模板
- [ ] 批量操作
- [ ] 导出功能

继续，写用户故事。
```

### 第五步：用户故事

```
用户：作为 AI 助手，我想快速搜索相关文档，以便回答问题。

**搜索功能**:
> 作为 AI 助手
> 我想快速搜索相关文档
> 以便准确回答用户问题

**验收标准**:
- [ ] 搜索响应时间 < 100ms
- [ ] 支持关键词匹配
- [ ] 支持模糊搜索
- [ ] 返回结果按相关性排序

**文档创建**:
> 作为 AI 助手
> 我想创建新的知识文档
> 以便分享学到的新知识

**验收标准**:
- [ ] 支持 Markdown 格式
- [ ] 自动提取标签
- [ ] 自动记录创建时间
- [ ] 创建后立即可搜索

继续，非功能需求有哪些？
```

## 💡 使用技巧

### 技巧 1: 先探索再写 PRD
- 先用 `brainstorming` 探索需求
- 明确后再写 PRD，避免返工

### 技巧 2: 验收标准要具体
- 用数字量化（如 <100ms）
- 可测试、可验证
- 参考 `HONEST_VERIFICATION_GUIDELINES.md`

### 技巧 3: 优先级分明
- P0 是 MVP，必须完成
- P1 是重要，可以延后
- P2 是锦上添花

### 技巧 4: 关联后续流程
- PRD 完成后用 `/plan-launch` 规划上线
- 用 `planning-with-files` 创建任务
- 用 `programming-workflow` 开发
