# 枫林工作流 - 快速开始指南

> ⏱️ 5分钟上手，立即开始高效开发

---

## 🎯 一句话概括

**枫林工作流 = 理解 → 规划 → 执行 → 沉淀**

根据项目复杂度，自动调整每个阶段的深度。

---

## 🚀 5分钟快速上手

### 第1分钟：判断项目复杂度

| 预计时间 | 复杂度 | 处理方式 |
|---------|--------|---------|
| < 2小时 | 🟢 简单 | 口头确认，直接执行 |
| 2小时-2天 | 🟡 中等 | 简要规划，然后执行 |
| > 2天 | 🔴 复杂 | 详细规划，确认后执行 |

**快速判断：**
```
问自己：这个项目我需要做多久？
- 几小时 → 简单
- 一两天 → 中等
- 好几天 → 复杂
```

---

### 第2分钟：选择规划方式

#### 🟢 简单项目 — 口头规划

```
"这是一个简单的功能，我计划：
1. 修改 xxx 文件
2. 添加 yyy 功能
3. 测试验证
预计 30 分钟完成，可以吗？"
```

**不需要文档，直接干！**

#### 🟡 中等项目 — 简要规划

在 `memory/daily/YYYY-MM-DD.md` 写几行：

```markdown
## 项目：用户登录功能
- 目标：实现用户登录
- 步骤：1.UI 2.后端 3.测试
- 预计：2小时
```

#### 🔴 复杂项目 — 详细规划

创建 `docs/plans/YYYY-MM-DD-feature.md`：

```markdown
# 项目规划：重构消息系统

## 目标
实现新的消息推送系统

## 技术方案
- 使用 WebSocket
- Redis 做消息队列

## 步骤
1. 搭建 WebSocket 服务
2. 集成 Redis
3. 前端对接
4. 测试上线

## 预计：3天
```

---

### 第3-4分钟：执行开发

#### 标准执行流程

```
1. 环境准备
   ├─ 检查分支
   ├─ 确认依赖
   └─ 准备环境

2. 编码实现
   ├─ 小熊-代码主责
   └─ 复杂模块 spawn 子 Agent

3. 质量把关（按需）
   ├─ 安全相关 → security-sentinel
   ├─ 性能关键 → performance-oracle
   └─ 常规项目 → 自我审查

4. 测试验证
   ├─ 基本功能
   └─ 边界情况
```

#### 什么时候需要审查？

| 情况 | 需要审查？ | 谁审查 |
|------|----------|--------|
| 涉及用户数据/支付 | ✅ 必须 | security-sentinel |
| 性能关键路径 | ✅ 推荐 | performance-oracle |
| 架构重大变更 | ✅ 推荐 | architecture-strategist |
| 普通功能开发 | ❌ 跳过 | 自我审查 |
| Bug 修复 | ❌ 跳过 | 自我审查 |

---

### 第5分钟：自动沉淀

**项目完成后，自动记录：**

```markdown
## 项目总结：用户登录功能
- 时间：1.5小时
- 结果：✅ 成功
- 关键决策：使用 JWT 而非 Session
- 遇到的问题：跨域配置
- 解决方案：配置 CORS 允许 credentials
- 可复用：JWT 工具函数
```

**自动保存到** `memory/daily/YYYY-MM-DD.md`

## 💡 常用命令速查

### 文件操作

```bash
# 查看今日工作记录
cat memory/daily/$(date +%Y-%m-%d).md

# 创建项目规划文件
touch docs/plans/$(date +%Y-%m-%d)-feature.md

# 创建解决方案记录
touch docs/solutions/$(date +%Y-%m-%d)-problem.md
```

### Git 操作

```bash
# 创建功能分支
git checkout -b feature/login

# 提交（遵循规范）
git commit -m "feat: add user login"

# 推送
git push origin feature/login
```

### Agent 调用

```bash
# 代码任务 → 小熊-代码
# 写作任务 → 小熊-运营
# 研究任务 → 小熊-研究
# 决策任务 → 小熊-统筹
```

## 🆘 遇到问题？

| 问题 | 解决方案 |
|------|---------|
| 不知道项目复杂度 | 先按中等处理，执行中发现简单了再调整 |
| 规划时间太长 | 规划时间应 < 10% 项目时间，太长了就简化 |
| 不确定要不要审查 | 涉及安全/性能/架构就审查，其他跳过 |
| 知识沉淀太麻烦 | 让统筹 Agent 自动处理，你只管执行 |
