# 四层记忆架构

> 让 AI Agent 拥有分层记忆能力，优化 Token 消耗

## 架构概览

```
┌─────────────────────────────────────────────────────┐
│  Layer 1: 上下文记忆（Context Memory）              │
│  - 当前对话、系统提示、工具结果                      │
│  - 特点：临时、有限、每次消耗 Token                  │
└─────────────────────────────────────────────────────┘
                         ↓
┌─────────────────────────────────────────────────────┐
│  Layer 2: 会话记忆（Session Memory）                │
│  - 本次会话的完整记录                                │
│  - 特点：会话内持久、自动压缩                        │
└─────────────────────────────────────────────────────┘
                         ↓
┌─────────────────────────────────────────────────────┐
│  Layer 3: 工作记忆（Working Memory）                │
│  - 近期日志、临时笔记                                │
│  - 特点：按日归档、快速检索                          │
└─────────────────────────────────────────────────────┘
                         ↓
┌─────────────────────────────────────────────────────┐
│  Layer 4: 长期记忆（Long-term Memory）              │
│  - 核心知识、用户偏好、重要决策                      │
│  - 特点：永久存储、语义检索                          │
└─────────────────────────────────────────────────────┘
```

## Layer 1: 上下文记忆

### 组成

```
Context = System Prompt + Conversation History + Tool Results + Attachments
```

### 特点

| 特性 | 说明 |
|------|------|
| 临时性 | 只在当次对话有效 |
| 有限性 | 受模型窗口限制 |
| 成本高 | 每次都要消耗 Token |

### 管理

- **压缩机制**：接近窗口限制时自动压缩
- **裁剪策略**：截断冗长日志，保留关键信息
- **缓存优化**：重复前缀只需 10% Token

## Layer 2: 会话记忆

### 组成

```
Session = 完整对话历史 + 工具调用记录 + 压缩摘要
```

### 管理

- **自动压缩**：上下文达到 75% 时触发
- **记忆刷写**：压缩前存储重要信息
- **摘要保存**：存入会话转录文件

### 配置示例

```json
{
  "compaction": {
    "reserveTokensFloor": 20000,
    "memoryFlush": {
      "enabled": true,
      "softThresholdTokens": 40000,
      "prompt": "将需要长期记录的内容写入 memory/YYYY-MM-DD.md"
    }
  }
}
```

## Layer 3: 工作记忆

### 文件结构

```
memory/
├── 2026-03-13.md    # 今日日志
├── 2026-03-12.md    # 昨日日志
├── 2026-03-11.md    # 前日日志
└── ...
```

### 内容示例

```markdown
# 2026-03-13

## 上午 10:30 - API 讨论
与用户探讨了 REST 和 GraphQL 的选型，最终决定采用 REST 方案。

## 下午 14:15 - 版本部署
将 v2.3.0 版本部署至生产环境，全程无异常。

## 下午 16:00 - 用户偏好
用户表示相比 JavaScript，更倾向使用 TypeScript。
```

### 读取规则

每次会话开始时：
1. 读取今日日志
2. 读取昨日日志
3. 通过检索获取更早的相关内容

## Layer 4: 长期记忆

### 文件结构

```
MEMORY.md              # 核心知识库
memory/projects/       # 项目记忆
  ├── project-a/
  │   ├── index.md
  │   └── tasks.md
  └── project-b/
      └── index.md
memory/core/           # 核心记忆
  ├── lessons.md       # 经验教训
  └── decisions.md     # 重要决策
```

### 内容示例

```markdown
# 长期记忆

## 用户偏好
- 编程语言：更倾向 TypeScript
- 内容风格：喜欢简洁的解释

## 核心联系人
```

## 层级交互

```
用户提问
    │
    ▼
┌─────────────┐
│ 检查 Layer 1│ ← 上下文中是否有答案？
└──────┬──────┘
       │ 无
       ▼
┌─────────────┐
│ 检索 Layer 3│ ← 工作记忆中是否有相关记录？
└──────┬──────┘
       │ 无
       ▼
┌─────────────┐
│ 检索 Layer 4│ ← 长期记忆中是否有相关知识？
└──────┬──────┘
       │
       ▼
   回答用户
       │
       ▼
┌─────────────┐
│ 写入记忆    │ ← 根据重要性写入 Layer 3 或 4
└─────────────┘
```

## Token 消耗对比

| 场景 | 传统方案 | 四层架构 |
|------|---------|---------|
| 启动加载 | 2000 Token | 200 Token（索引） |
| 记忆检索 | 全量读取 | 200 Token（片段） |
| 会话维护 | 线性增长 | 自动压缩 |
| 长期存储 | 有限 | 无限 |

## 最佳实践

### 1. 分级存储

- 临时笔记 → Layer 3（按日归档）
- 重要决策 → Layer 4（长期记忆）
- 技能经验 → 知识库（技能文件）

### 2. 定期维护

- 每周回顾 Layer 3，提取重要内容到 Layer 4
- 每月归档旧日志
- 定期更新索引

### 3. 检索优先

- 不要全量加载记忆
- 先检索，后读取
- 使用混合检索提高准确率
