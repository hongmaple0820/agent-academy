# AI 开发工作流规范

> **版本**：v1.0.0  
> **状态**：已发布  
> **创建日期**：2026-03-23  
> **最后更新**：2026-03-23  
> **适用范围**：所有 AI Agent 及人类开发者  
> **来源**：提炼自 WestWorld 项目实战经验

---

## 概述

本文档定义了一套"Zero-Bug Delivery"开发工作流，确保 AI Agent 和人类开发者产出的代码能够**安全、完整、可维护**地上线，避免"Vibe Coding 90% 综合症"（只写主流程，忽视边界和运维细节）。

### 核心原则

| 原则 | 说明 |
|------|------|
| **流程重于灵感** | Demo 靠灵感，上线靠流程。必须严格遵循四步工作流 |
| **契约先行** | 编写逻辑前，必须明确验收标准和边界条件 |
| **最小闭环** | 每次改动必须是原子性的、可验证的，确保功能完整 |
| **专注原则** | 一次对话只解决一类问题，严禁混合修改 |

## Phase 1：需求冻结与契约定义

在开始编码前，必须以注释形式在代码顶部列出**验收契约**：

### 契约要素

```typescript
/**
 * ============================================
 * 验收契约 (Acceptance Contract)
 * ============================================
 * 功能边界：
 *   - 支持：用户名密码登录、手机号验证码登录
 *   - 不支持：第三方 OAuth 登录（下期规划）
 *
 * 异常处理（至少 3 种）：
 *   - E001: 用户不存在 → 404 + "账号未注册"
 *   - E002: 密码错误（连续 5 次）→ 429 + 账号锁定 30 分钟
 *   - E003: 验证码过期 → 410 + "请重新获取"
 *
 * 日志规范：
 *   - INFO: 登录成功（脱敏手机号）
 *   - WARN: 连续失败 3 次
 *   - ERROR: 系统异常（记录完整 stack）
 *
 * 回滚条件：
 *   - 事务内操作失败时全部回滚
 *   - Redis token 写入失败不阻断登录（降级为无状态）
 * ============================================
 */
```

### 检查清单

- [ ] 功能边界明确（支持什么、不支持什么）
- [ ] 至少定义 3 种异常处理场景
- [ ] 日志级别和数据脱敏要求已确认
- [ ] 数据变更场景的回滚策略已明确

## Phase 2：最小闭环执行

### 执行顺序

```
1. 主流程 (Happy Path)
   └─ 确保核心逻辑通畅

2. 单元测试
   └─ 覆盖边界情况和异常场景

3. Lint & Diff
   └─ 代码规范检查 + 改动摘要

4. 自测验证
   └─ 手动或自动运行确认
```

### 代码编写规范

#### 引用代码时标注位置

```
file_path:line_number
例：src/api/auth.ts:42
```

#### 命名规范

| 类型 | 规范 | 示例 |
|------|------|------|
| 组件文件 | PascalCase | `FilePreview.vue` |
| API 函数 | camelCase | `getUserInfo` |
| 类型/接口 | PascalCase | `UserInfoResponse` |
| 常量 | SCREAMING_SNAKE_CASE | `API_BASE_URL` |
| CSS 类 | kebab-case | `file-list-item` |
| 私有成员 | 下划线前缀 | `_internalState` |

#### 注释规范

```typescript
/**
 * 函数/类说明
 * 详细描述功能
 * @version v1.0.0
 * @param paramName - 参数说明
 * @returns 返回值说明
 */
```

### 测试要求

- **必须**提供测试用例（至少测试思路）
- **必须**覆盖边界情况和异常场景
- 不能只有 Happy Path 测试

## Phase 3：上线护栏

### 依赖引入

引入新库时必须：

```markdown
- 库名：lodash
- 版本：^4.17.21
- 用途：深拷贝工具函数
- 冲突风险评估：无，项目已有相同版本
- 替代方案：可用 structuredClone() 替代（Chrome 98+）
```

### 环境配置

关键配置需增加启动健康检查：

```typescript
// 数据库连接健康检查
app.listen(port, async () => {
  try {
    await db.query('SELECT 1');
    logger.info('Database connected');
  } catch (error) {
    logger.error('Database connection failed', error);
    process.exit(1); // 启动失败，快速失败
  }
});
```

### 危险操作确认

遇到以下操作时，**必须**使用 `⚠️ WARNING` 标签并提示二次确认：

```
⚠️ WARNING: 即将执行 DELETE 操作
表：users | 条件：created_at < '2025-01-01'
影响行数：约 1,200 行
回滚方案：已备份至 backup_users_20260323.sql
确认执行？(y/N)
```

覆盖范围：`DELETE`、`DROP`、`rm -rf`、`TRUNCATE`、`UPDATE（批量）`

## Phase 4：发布前检查清单

每次代码输出末尾，生成完整的发布检查：

### 五维度检查

```markdown
## 发布检查清单

- ✅ **性能**：是否存在 N+1 查询、死循环或内存泄漏风险？
- ✅ **监控**：关键异常是否已添加监控埋点和告警机制？
- ✅ **回滚**：是否有对应的回滚方案（SQL 脚本或回滚代码）？
- ✅ **安全**：是否存在 SQL 注入、XSS 或其他安全风险？
- ✅ **兼容性**：是否兼容现有系统和数据结构？
```

### 额外检查

- [ ] 代码通过类型检查（TypeScript strict mode）
- [ ] 代码通过构建
- [ ] 新增文件已正确分类（temp/ vs 生产目录）
- [ ] 临时文件已清理或放入 `temp/`
- [ ] 文档/脚本版本号已更新
- [ ] Commit message 符合规范

## 禁止行为清单

以下行为在任何情况下都是**不允许**的：

| 禁止行为 | 说明 | 正确做法 |
|----------|------|----------|
| ❌ 裸奔代码 | 生成没有错误处理的代码 | 所有外部调用加 try-catch |
| ❌ 偷偷引入新库 | 未声明的情况下引入第三方依赖 | 先评估并在契约中说明 |
| ❌ 只写代码不写测试 | 提交代码但不提供测试 | 至少提供测试思路和用例 |
| ❌ 越界修改 | 用户说"只改 A"却碰了 B | 严格遵循修改范围 |
| ❌ 提交敏感信息 | 代码中包含密码、API Key | 使用环境变量或密钥管理 |
| ❌ 无版本号文档 | 提交文档/脚本不标注版本 | 遵循 `v{主}.{次}.{修订}` 格式 |
| ❌ 危险操作不确认 | 直接执行 DELETE/DROP | 先展示影响范围，二次确认 |

## 文件放置决策树

AI 在生成任何文件前，按以下决策流程放置：

```
开始
  │
  ├─ 临时测试/调试/草稿？
  │    └─ 是 → temp/script/ 或 temp/draft/（不提交 Git）
  │
  ├─ 数据库脚本？
  │    ├─ 表结构定义 → script/sql/ddl/
  │    ├─ 数据操作   → script/sql/dml/
  │    └─ 迁移脚本   → script/sql/migration/
  │
  ├─ 部署脚本？
  │    └─ 是 → script/deploy/
  │
  ├─ 对外文档（API/指南/FAQ）？
  │    └─ 是 → docs/open/{api|guide|faq}/
  │
  ├─ 内部技术文档？
  │    └─ 是 → docs/internal/{architecture|decisions|specs}/
  │
  ├─ 计划/需求文档？
  │    └─ 是 → docs/plans/{roadmap|requirements|milestones}/
  │
  └─ 源代码？
       └─ 是 → src/
```

## 文档与脚本元数据标准

### 文档元数据（必须包含）

```markdown
# 文档标题

> 版本：v1.0.0
> 状态：草稿 | 评审中 | 已批准 | 已废弃
> 创建日期：YYYY-MM-DD
> 最后更新：YYYY-MM-DD
```

### 脚本头部（生产脚本必须包含）

```sql
-- ============================================================
-- 脚本名称：001_create_user_table_v1.0.0.sql
-- 版本：v1.0.0
-- 用途：创建用户表及索引
-- 执行顺序：001
-- 前置条件：无
-- 回滚脚本：script/sql/ddl/001_drop_user_table_v1.0.0.sql
-- 创建日期：2026-03-23
-- 作者：作者名
-- ============================================================

START TRANSACTION;

-- 脚本内容

COMMIT;
```

### 脚本命名规范

| 类型 | 格式 | 示例 |
|------|------|------|
| 初始化 | `{模块}_v{版本}.sql` | `user_v1.0.sql` |
| 升级 | `{模块}_upgrade_v{版本}.sql` | `user_upgrade_v1.1.sql` |
| 补丁 | `{模块}_patch_{日期}_{序号}.sql` | `user_patch_20260323_01.sql` |
| 序号式 | `{序号}_{描述}_v{版本}.{ext}` | `001_create_user_v1.0.0.sql` |

### 版本号规范

采用语义化版本号 `v{MAJOR}.{MINOR}.{PATCH}`：

| 变更类型 | 版本升级 | 示例 |
|----------|----------|------|
| 不兼容的 API 修改 | MAJOR +1 | v1.0.0 → v2.0.0 |
| 向下兼容的新功能 | MINOR +1 | v1.2.0 → v1.3.0 |
| 向下兼容的问题修复 | PATCH +1 | v1.2.3 → v1.2.4 |

## Git 提交规范

### Commit Message 格式

```
<type>(<scope>): <subject>

<body>
```

### 类型说明

| 类型 | 说明 | 示例 |
|------|------|------|
| `feat` | 新增功能 | `feat(auth): 新增手机号验证码登录` |
| `fix` | 修复 bug | `fix(api): 修复分页参数未校验导致崩溃` |
| `docs` | 文档更新 | `docs(guide): 更新开发工作流规范` |
| `refactor` | 代码重构 | `refactor(utils): 提取公共校验函数` |
| `test` | 测试相关 | `test(auth): 添加登录失败边界测试` |
| `chore` | 构建/工具/配置 | `chore(deps): 升级 vite 到 5.2` |
